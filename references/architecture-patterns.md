# Skill Architecture Patterns

## 前置判断：你的 Skill 是"大项目"吗？

不是所有 skill 都需要复杂架构。以下评分卡帮助判断：

### 复杂度评分卡

每项符合计 1 分：

| # | 信号 | 说明 |
|---|------|------|
| 1 | **体量信号** | SKILL.md 预计超过 200 行 |
| 2 | **子任务信号** | 包含 5+ 个相对独立的子任务（每个子任务有自己的输入/处理/输出） |
| 3 | **外部依赖信号** | 需要调用 MCP 工具、外部 API、或查询动态数据源 |
| 4 | **状态信号** | 步骤间有数据依赖（步骤 B 的输入来自步骤 A 的输出），需要跨步骤追踪进度 |
| 5 | **跨轮次信号** | 单次对话无法完成全部工作（需要多轮交互、或中途可能被 compact/中断） |
| 6 | **输出信号** | 产出结构化长报告（HTML/Markdown/PDF）、多文件组合输出、或需要增量追加 |

### 评分结果

| 得分 | 复杂度 | 推荐架构 |
|------|--------|---------|
| 0-1 | **简单** | 单文件 `SKILL.md` 即可，不需要拆分 |
| 2-3 | **中等** | `SKILL.md` + `resources/`（参考文档、模板脚本），不需要编排-执行分离 |
| 4-6 | **大型** | 需要使用本文档中的架构模式：编排-执行分离 + 文件状态机 + 渐进式输出 |

### 大型 Skill 的典型特征画像

如果你在需求阶段就发现以下特征，大概率是大型 skill：

- **"帮我审计 X"** → 暗示多步骤检查、动态数据查询、结构化报告输出 → 通常 4-6 分
- **"生成一份完整的 Y 报告"** → 暗示多章节、数据聚合、上下文一致性校验 → 通常 4-5 分
- **"从 A 系统迁移到 B 系统"** → 暗示步骤依赖、状态追踪、回滚处理 → 通常 5-6 分
- **"帮我写一个 Z"** → 单步、无外部依赖 → 通常是简单 skill（0-1 分）

### 判断原则

**宁可一开始简单，在迭代中升级。** 不要在需求阶段就过度设计——第一版用单文件验证可行性，当体量/复杂度自然增长到触发上述信号时，再按本文档模式重构。

---

以下模式适用于评分 4+ 的大型 skill。

## 1. 编排-执行分离（Orchestration-Execution Separation）

**问题**：一个 skill 包含多个独立子任务，全部写在一个 SKILL.md 中导致文件过长、修改一个子任务影响全局、上下文容易膨胀。

**方案**：将编排逻辑（做什么、按什么顺序做）与执行逻辑（每个子任务怎么做）分离到不同文件。

```
skill-name/
├── SKILL.md              # 编排层：触发条件、主流程、资源索引
└── resources/
    └── tasks/             # 执行层：每个子任务独立目录
        ├── task_01/
        │   ├── instructions.md   # 该任务的执行指令
        │   └── template.sql      # 该任务的 SQL/代码模板
        ├── task_02/
        └── ...
```

**SKILL.md 只做三件事：**
1. 声明默认参数和全局约束
2. 定义编排循环（逐个执行任务、处理异常、追踪进度）
3. 指向执行层的资源文件

**关键规则**：编排层不写具体执行细节，执行层的每个任务目录自包含、可独立修改。

**实际案例**：dataphin-account-permission-audit skill 将 10 个审计检查项从单体 sql_templates.md 拆分为 10 个独立 check_items/ 目录，每个包含 `sql.sql` + `query_rules.md` + `think_rules.md`。

## 2. 文件状态机（File State Machine）

**问题**：Claude Code 的 auto-compact 会在上下文接近 token 上限时自动压缩对话历史。如果中间结果只存在对话上下文中，压缩后就丢失了。

**方案**：每个步骤完成后，将结果写入持久化文件。编排循环从文件读取进度，而非依赖对话记忆。

```
workspace/
├── state.json          # 进度追踪：{completed: ["01","02"], pending: ["03",...]}
├── task_01_result.json # 步骤 1 的输出
├── task_02_result.json # 步骤 2 的输出
└── ...
```

**编排循环伪代码：**
```
for task in task_list:
  if task in state.completed: skip
  result = execute(task)
  write result to task_XX_result.json
  state.completed.append(task)
  write state to state.json
```

**Compact 恢复规则（必须在 SKILL.md 中写明）：**
> 如果经历 compact/会话恢复，继续前必须先重新读取 `task_XX_result.json` 和 `state.json`，以本地文件内容为唯一数据源。禁止仅基于压缩摘要或记忆继续推断。

---

## 3. 渐进式报告追加（Incremental Report Assembly）

**问题**：等所有步骤完成后再生成最终输出，存在两个风险：① 上下文在最后阶段被压缩导致早期结果丢失；② 用户等待时间长，看不到中间进展。

**方案**：每完成一个步骤，立即将其结果追加到最终输出文件中。

```
step_1 完成 → 追加到 report.html
step_2 完成 → 追加到 report.html
...
全部完成 → 在 report.html 最前面插入摘要，末尾追加闭环
```

**关键约束**：
- 每个步骤追加的内容应自包含（不依赖前面步骤的上下文）
- 最终汇总阶段重新读取完整输出文件，进行总体数据一致性检查
- 适用于 HTML、Markdown 等可增量追加的格式

---

## 4. 总分总报告结构（Overview-Detail-Review）

**问题**：长篇报告如何组织才能让读者快速把握全貌又能深入细节？

**方案**：三层结构：

```
总① → 审计总览（参数、风险概览、核心发现摘要）
 分 → 各检查项详情（每个检查项自包含：范围-目标-程序-发现）
总② → 总体复核闭环（数据一致性、穷尽列示、幻觉检查）
```

**总①**在全部"分"完成后插入到最前面（结论优先），**总②**在总①插入后进行。

---

## 5. 查询-分析约束分离（Query-Analysis Constraint Separation）

**问题**：一个复杂步骤包含"查什么数据"和"怎么分析数据"两个不同阶段，耦合在一起导致：① compact 可能发生在查询后、分析前；② 修改分析规则时容易误改查询逻辑。

**方案**：每个任务目录包含两个独立文件：

| 文件 | 加载时机 | 内容 |
|------|---------|------|
| `query_rules.md` | 执行查询前 | 数据表、过滤条件、分页规则、异常检测 |
| `think_rules.md` | 分析结果前 | 分类规则、风险定级、输出格式、折叠规范 |

**分离理由**：SQL 执行和结果分析之间可能发生 compact。分离后每个文件独立加载，互不耦合。

---

## 6. 自动申请-遇阻才问（Auto-Apply, Ask on Failure）

**问题**：skill 启动时需要申请 MCP 权限、创建目录、确认输出路径等前置操作。如果每一步都停下来问用户，用户体验差、无法无人值守运行。

**方案**：将可自动化的前置操作全部自动执行，仅在失败时阻断询问。

**模式**：
```
前置操作 → 自动执行（不询问）
  ├── 全部成功 → 用户零感知，继续主流程
  └── 部分失败 → 非阻断式汇总报告："已自动完成 N 项，M 项失败需手动处理"
```

**适用场景**：
- MCP 权限申请
- 输出目录创建
- 默认参数确定（日期、范围、路径）
- 依赖检查

**不适用**：涉及安全边界（如删除操作、提权、跨租户访问）的前置操作仍需确认。

---

## 何时使用这些模式

| 信号 | 推荐模式 |
|------|---------|
| SKILL.md > 300 行 | 编排-执行分离 |
| 5+ 个独立子任务 | 编排-执行分离 + 文件状态机 |
| 依赖 MCP 查询外部数据 | 文件状态机 + 查询-分析约束分离 |
| 输出长篇结构化报告 | 渐进式报告追加 + 总分总结构 |
| 需要用户授权的引导步骤 | 自动申请-遇阻才问 |

---

## 反模式（应避免）

- **单文件巨无霸**：所有逻辑塞在 SKILL.md，超过 500 行不拆分
- **上下文依赖链**：步骤 N 的结果只在对话上下文中，不写文件，compact 后丢失
- **过度询问**：每个默认参数都问用户，实际 99% 场景用默认值
- **一次性输出**：等所有步骤完成后才生成报告，中间无持久化
